Kiwix.org - Case Study (2020)

Technicality level: 4/5

About the client

Kiwix lets you access free knowledge - even offline. You can store Wikipedia, Project Gutenberg, TED Talks or any website on your mobile phone or computer. Kiwix makes knowledge available to people with no or limited internet access. The software as well as the content is free to use for anyone.

The challenge

Kiwix makes online knowledge available to the parts of the world without internet. In order to do that, Kiwix needs to collect data from Wikipedia, write the data in the correct format and store in the library. However, the process is not that simple. Downloading those pages requires a lot of technical knowledge. Some pages might consist of text only, but most have photos or even videos. Thus, Kiwix needs an efficient process that will make it easier to scale and grow the library.

English Wikipedia right now has more than 6 million articles with an average of 592 words in each article.

Kiwix used two separate programs in order to download and store articles from Wikipedia. The first one was in Python. It downloaded the content and stored it to the hard drive. The second one was in C++. It read content from the hard drive (used the libzim library) and bundled the Zim file. Only after this 3-step process finished, the content was ready for the library.

The problem was these two programs did not communicate with each other because they were programmed in different languages. It made the process inefficient.

Original Workflow

Project goal

Our goal was to think of a new workflow that would make it possible for those two programs in Python and C++ communicate. It would make the process more efficient and scalable. We found the best way for the two programs to communicate would be to create a wrapper.

New Workflow

Scope of work

  • Working python-libzim PyPI package for sotoki that doesn’t use zimwriterfs
  • pip install … process that works across Linux, and Docker (macOS not yet)
  • C++ compilation process that works seamlessly in Docker
  • Tests that build & run in CI automatically upon pull requests / merges (+release if needed) (just for the bindings, not all of libzim)
  • Python binding library refactoring and implementation
  • https://github.com/openzim/python-libzim/
  • Documentation for both end-users and developers
  • Update sotoki to work with the new python-libzim (update from zimwriterfs)

Time & team:

  • 4 weeks of full-time work with 1 senior developer on the project.

Technical stack:

  • Python
  • C++
  • Cython

Final Deliverables

We created a Class Blend Wrapper. Now, there is no need to write intermediate content to the hard drive and there is no need to create copies between Python and C++.
We created a Class Blend Wrapper that makes the process more efficient. It passes an interface implemented in Python instead of creating unnecessary copies.
